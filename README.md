## Описание проекта

Данный проект представляет собой учебную реализацию защиты веб-приложения от атак CSRF (Cross-Site Request Forgery) и XSS (Cross-Site Scripting) с использованием безопасных cookies.

Проект состоит из двух микросервисов:

- Auth Service — отвечает за аутентификацию пользователей и выдачу secure cookies
- Tasks Service — управляет задачами пользователей с применением CSRF-защиты и санитизацией ввода

Архитектура включает Nginx в качестве reverse proxy с TLS-терминацией и PostgreSQL для хранения данных.

## Цель работы

Научиться безопасно использовать cookies в серверном приложении и внедрить практические меры защиты от CSRF и XSS.

Планируемые результаты:

- Понимание связи cookies с CSRF-уязвимостями
- Настройка cookies с флагами HttpOnly, Secure, SameSite
- Реализация CSRF-защиты на основе паттерна Double Submit Cookie
- Применение базовых мер защиты от XSS (экранирование и заголовки безопасности)

### Пошаговая установка

```bash

git clone https://github.com/ybotet/pz6_csrf_xss.git
cd pz6_csrf_xss

# 2. Генерация SSL-сертификатов (если не существуют)
cd deploy/tls
openssl req -x509 -newkey rsa:2048 -nodes \
  -keyout key.pem -out cert.pem -days 365 \
  -subj "//CN=localhost"


# 3. Запуск всех сервисов
docker-compose up -d --build

# 4. Проверка статуса контейнеров
docker-compose ps

# 5. Просмотр логов
docker-compose logs -f
```

## Контрольные вопросы

1. Почему CSRF возможен при использовании cookies?

CSRF становится возможным потому что браузер автоматически прикрепляет cookies к запросам на ваш домен, даже если запрос инициирован с вредоносной страницы. Злоумышленник может заставить браузер жертвы отправить запрос, который будет выглядеть как запрос аутентифицированного пользователя, так как cookies будут отправлены автоматически, и сервер не сможет отличить легитимный запрос от поддельного.

2. Что делает флаг SameSite и какие есть режимы?

Флаг SameSite ограничивает отправку cookies при кросс-сайтовых запросах. Режим Lax отправляет cookie только при навигации (переход по ссылке), но не при вложенных запросах — это разумный минимум для большинства сценариев. Режим Strict блокирует отправку при любых кросс-сайтовых запросах, обеспечивая максимальную безопасность. Режим None разрешает отправку всегда, но требует флаг Secure и используется только когда необходимо third-party поведение.

3. Чем HttpOnly защищает от XSS и почему он не "лечит" XSS полностью?

HttpOnly запрещает доступ к cookie из JavaScript, что защищает сессионную cookie от кражи при XSS-атаке. Однако HttpOnly не решает проблему XSS полностью, потому что злоумышленник всё ещё может модифицировать содержимое страницы, отправлять запросы от имени пользователя (используя автоматически отправляемые cookies), красть другие данные со страницы или перенаправлять пользователя на фишинговые сайты.

4. Почему Secure обязателен, если cookie несёт сессию?

Флаг Secure обязателен для сессионных cookies, потому что защищает от перехвата при передаче по незащищённому HTTP-соединению (атака Man-in-the-Middle). Если cookie передаётся по HTTP, злоумышленник в общедоступной сети WiFi может легко перехватить её и получить доступ к учётной записи пользователя.

5. Как работает double-submit CSRF защита?

Сервер генерирует случайный токен и отправляет его клиенту в cookie csrf_token (без HttpOnly, чтобы JavaScript мог его прочитать). Клиент читает значение этой cookie и добавляет его в заголовок X-CSRF-Token при каждом запросе, изменяющем состояние. Сервер сравнивает токен из cookie и из заголовка — если они совпадают, запрос считается легитимным. Вредоносный сайт может заставить браузер отправить cookie, но не может прочитать её из-за Same Origin Policy, поэтому не может сформировать правильный заголовок.

6. Что такое XSS и какие базовые меры защиты применимы на backend?

XSS (Cross-Site Scripting) — это атака, при которой злоумышленник внедряет вредоносный скрипт в веб-страницу, который затем выполняется в браузере других пользователей. Базовые меры защиты на backend включают: экранирование HTML-символов при выводе пользовательского ввода, валидацию входных данных, установку заголовка Content-Security-Policy, использование HttpOnly для сессионных cookies и санитизацию HTML, если разрешено форматирование.
